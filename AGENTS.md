- [2026-03-25] Repo package manager: This repo is configured to use `npm`, not `pnpm`, and its Vitest setup does not accept Jest-style `--runInBand`. When running tests here, future agents should use `npm test` or documented npm scripts.
- [2026-03-25] Web env files after workspace move: When the Next app lives under `apps/web`, Next only auto-loads env files from that directory. Future agents should keep `apps/web/.env`, `.env.example`, and `.env.production` symlinked to the repo root copies so web dev/build keeps seeing the existing Clerk and DB configuration.
- [2026-03-25] Expo workspace commands: In this repo, Expo behaves correctly when run with `npm exec -w @lift-prog/mobile -- expo ...`, and Expo SDK 55 monorepo resolution worked with the default `expo/metro-config`. Future agents should avoid custom Metro `nodeModulesPaths` overrides here because they broke React Native dependency resolution during iOS bundling.
- [2026-03-25] Expo mobile entrypoint: In this workspace, `apps/mobile` should use a local entry file such as `index.ts` with `registerRootComponent(App)` instead of `main: "expo/AppEntry"`. After npm hoisting changed, `expo/AppEntry` resolved `../../App` from the wrong place and broke `expo export`.
- [2026-03-25] Mobile env loading: Expo only auto-loads env files from `apps/mobile`, so future agents should keep `apps/mobile/.env` and `apps/mobile/.env.production` symlinked to the repo root env files. `apps/mobile/app.config.ts` now reads the root Clerk publishable key through those symlinks and exposes it to the app.
- [2026-06-20] Local-first mobile data direction: The future iOS app may use local SQLite as the primary database with no required login, but future schema and repository work must preserve a path for optional cloud backup/restore and multi-device sync. When changing IDs, migrations, persistence, or import/export flows, prefer stable client-generated IDs, append-only migrations, and repository boundaries that can later support cloud restore or sync without making the workout UI depend on the network.
- [2026-06-22] Agent-operable iOS app: A core tenant of the iOS app is that agents must be able to fully control and use it in both Simulator and iPhone Mirroring. When building mobile UI, future agents should keep controls accessible, deterministic, visible to automation, and easy to exercise end-to-end so agents can autonomously iterate, find issues, and verify fixes quickly.
- [2026-06-22] Local iPhone dev build: For physical iPhone testing without paid Expo services, install `expo-dev-client`, run `npm exec -w @lift-prog/mobile -- expo run:ios --device <UDID>`, and keep Metro running with `npm exec -w @lift-prog/mobile -- expo start --dev-client --port 8081`. The generated iOS Podfile needed `use_modular_headers!` for `AppCheckCore`/`GoogleUtilities` Swift pod integration.
- [2026-06-22] Mobile tunnel workaround: `expo start --dev-client --tunnel` failed with `Cannot read properties of undefined (reading 'body')`, matching current Expo/ngrok tunnel instability. For gym testing, future agents should keep Metro on 8081 and use a separate Cloudflare tunnel to `http://localhost:8081`, then launch the installed dev client with a payload URL like `exp+lift-prog://expo-development-client/?url=<encoded tunnel URL>`.
